1 <section class="dex_guide"><h1 class="dex_title">Wangle</h1><section class="dex_document"><h1></h1><p class="dex_introduction">C++ networking library</p><p>Wangle is a library that makes it easy to build protocols, application clients, and application servers.</p>
2 
3 <p>It&#039;s like Netty + Finagle smooshed together, but in C++</p>
4 
5 ## Building and Installing
6 
7 The main dependencies are:
8 * The folly library from https://github.com/facebook/folly
9 * The fizz library from https://github.com/facebookincubator/fizz
10 * CMake
11 * OpenSSL, at least version 1.0.2+, preferably with TLS extension support.
12 
13 Once folly is installed, run the following inside the wangle directory to build, test, and install wangle:
14 ```
15 cmake .
16 make
17 ctest
18 sudo make install
19 ```

51 <h2 id="request-response-abstrac">Request / Response abstraction <a href="#request-response-abstrac" class="headerLink">#</a></h2>
52 
53 <p>This is roughly equivalent to the <a href="https://twitter.github.io/finagle/" target="_blank">Finagle</a> library.</p>
54 
55 <p>Aims to provide easy testing, load balancing, client pooling, retry logic, etc. for any request/response type service - i.e. thrift, http, etc.</p>
56 
57 <p>Service - a matched interface between client/server. A server will implement this interface, and a client will call in to it. These are protocol-specific</p>
58 
59 <p>ServiceFilter - a generic filter on a service. Examples: stats, request timeouts, rate limiting</p>
60 
61 <p>ServiceFactory - A factory that creates client connections. Any protocol specific setup code goes here</p>
62 
63 <p>ServiceFactoryFilter - Generic filters that control how connections are created. 95 <p><strong>channelFactory(ServerSocketFactory)</strong></p>
96 
97 <p>Sets up the type of server. Defaults to TCP AsyncServerSocket, but AsyncUDPServerSocket is also supported to receive udp messages. In practice, ServerBootstrap is only useful for udp if you need to multiplex the messages across many threads, or have TCP connections going on at the same time, etc. Simple usages of AsyncUDPSocket probably don&#039;t need the complexity of ServerBootstrap.</p>
98 
99 <p><strong>pipeline(PipelineFactory&lt;AcceptPipeline&gt;)</strong></p>
100 
101 <p>This pipeline method is used to get the accepted socket (or udp message) *before* it has been handed off to an IO thread. This can be used to steer the accept thread to a particular thread, or for logging.</p>
102 
103 <p>See also AcceptRoutingHandler and RoutingDataHandler for additional help in reading data off of the accepted socket <strong>before</strong> it gets attached to an IO thread. These can be used to hash incoming sockets to specific threads.</p>
104 
105 <p><strong>childHandler(AcceptorFactory)</strong></p>
106 
107 <p>Previously facebook had lots of code that used AcceptorFactories instead of Pipelines, this is a method to support this code and be backwards compatible. The AcceptorFactory is responsible for creating acceptors, setting up pipelines, setting up AsyncSocket read callbacks, etc.</p>

185 <h3 id="staticpipeline">StaticPipeline <a href="#staticpipeline" class="headerLink">#</a></h3>
186 
187 <p>Instantiating all these handlers and pipelines can hit the allocator pretty hard. There are two ways to try to do fewer allocations. StaticPipeline allows *all* the handlers, and the pipeline, to be instantiated all in the same memory block, so we only hit the allocator once.</p>
188 
189 <p>The other option is to allocate the handlers once at startup, and reuse them in many pipelines. This means all state has to be saved in the HandlerContext object instead of the Handler itself, since each handler can be in multiple pipelines. 227 <h2 id="services">Services <a href="#services" class="headerLink">#</a></h2>
228 
229 <p>A Pipeline was read() and write() methods - it streams bytes in one or both directions. write() returns a future, but the future is set when the bytes are successfully written. Using pipeline there is no easy way to match up requests and responses for RPC.</p>
230 
231 <p>A Service is an RPC abstraction - Both clients and servers implement the interface. Servers implement it by handling the request. Clients implement it by sending the request to the server to complete.</p>
232 
233 <p>A Dispatcher is the adapter between the Pipeline and Service that matches up the requests and responses. There are several built in Dispatchers, however if you are doing anything advanced, you may need to write your own.</p>
234 
235 <p>Because both clients and servers implement the same interface, mocking either clients or servers is trivially easy.</p>
236 
237 <h2 id="servicefilters">ServiceFilters <a href="#servicefilters" class="headerLink">#</a></h2>
238 
239 <p>ServiceFilters provide a way to wrap filters around every request and response. Things like logging, timeouts, retrying requests, etc. can be implemented as ServiceFilters.</p>
240 
241 <p>Existing ServiceFilters include:</p>
242 
243 <ul>
244 <li>CloseOnReleaseFilter - rejects requests after connection is closed. Often used in conjunction with</li>
245 <li>ExpiringFilter - idle timeout and max connection time (usually used for clients)</li>
246 <li>TimeoutFilter - request timeout time. Usually used on servers. Clients can use future.within to specify timeouts individually.</li>
247 <li>ExecutorFilter - move requests to a different executor.</li>
248 </ul>
